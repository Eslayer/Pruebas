API REST ENDPOINTS - QUICKBITE
===============================
Fecha: 17 de junio de 2026
Proyecto: Sistema de Pedidos de Comida QuickBite

INTRODUCCIÓN
============
Este documento describe los endpoints de la API REST de cada microservicio del sistema QuickBite para ser utilizados en Postman o Swagger.

CONFIGURACIÓN GENERAL
====================
Base URL: http://localhost:8080 (para desarrollo)
Content-Type: application/json
Authentication: Bearer Token (JWT)

1. SERVICIO DE AUTENTICACIÓN (quickbite-Auth-main)
==================================================
Port: 8081
Swagger UI: http://localhost:8081/swagger-ui.html

Endpoints:

POST /api/auth/register
Descripción: Registrar un nuevo usuario
Request Body:
{
  "username": "usuario",
  "email": "usuario@example.com",
  "password": "password123",
  "firstName": "Juan",
  "lastName": "Pérez",
  "phoneNumber": "1234567890",
  "role": "CLIENT"
}
Response: AuthenticationResponse

POST /api/auth/authenticate
Descripción: Autenticar usuario y obtener token
Request Body:
{
  "username": "usuario",
  "password": "password123"
}
Response: AuthenticationResponse

POST /api/auth/refresh
Descripción: Refrescar token de acceso
Request Body:
{
  "refreshToken": "token_de_refresco"
}
Response: AuthenticationResponse

POST /api/auth/validate
Descripción: Validar token de acceso
Headers: Authorization: Bearer {token}
Response: Boolean

GET /api/auth/profile/{id}
Descripción: Obtener perfil de usuario por ID
Headers: Authorization: Bearer {token}
Response: User

PUT /api/auth/profile/{id}
Descripción: Actualizar perfil de usuario
Headers: Authorization: Bearer {token}
Request Body:
{
  "firstName": "Juan",
  "lastName": "Pérez",
  "email": "usuario@example.com",
  "phoneNumber": "1234567890",
  "profileImage": "url_imagen",
  "address": "Dirección"
}
Response: User

CONFIGURACIÓN DE POSTMAN
=========================

1. Crear un nuevo Workspace en Postman
2. Crear una Collection llamada "QuickBite API"
3. Crear carpetas por servicio:
   - Authentication
   - Inventory
   - Menu
   - Orders
   - Kitchen
   - Notifications
   - Payments

4. Configurar variables de entorno:
   - base_url: http://localhost:8080
   - auth_port: 8081
   - inventory_port: 8082
   - menu_port: 8083
   - orders_port: 8084
   - kitchen_port: 8085
   - notifications_port: 8086
   - payments_port: 8087
   - token: (obtenido después de autenticación)

5. Configurar Pre-request Script para autenticación:
```javascript
if (pm.environment.get("token")) {
    pm.request.headers.add({
        key: 'Authorization',
        value: 'Bearer ' + pm.environment.get("token")
    });
}
```

6. Ejemplo de configuración de endpoint:
   - Method: POST
   - URL: {{base_url}}/api/auth/authenticate
   - Headers: Content-Type: application/json
   - Body: Raw JSON

AUTENTICACIÓN
=============
Para acceder a endpoints protegidos, primero debe autenticarse:

1. POST /api/auth/authenticate
2. Copiar el token del response
3. Guardarlo en la variable de entorno "token"
4. Usar el token en los headers: Authorization: Bearer {{token}}

CÓDIGOS DE RESPUESTA HTTP
===========================
200 OK: Solicitud exitosa
201 Created: Recurso creado exitosamente
400 Bad Request: Solicitud inválida
401 Unauthorized: No autorizado (token inválido o ausente)
403 Forbidden: Prohibido (sin permisos)
404 Not Found: Recurso no encontrado
500 Internal Server Error: Error del servidor

ERRORES COMUNES
===============
- 401 Unauthorized: Token expirado o inválido
- 404 Not Found: Endpoint o recurso no existe
- 500 Internal Server Error: Error en el servidor
- 503 Service Unavailable: Servicio no disponible (circuit breaker activado)

NOTAS
=====
- Todos los endpoints que requieren autenticación deben incluir el header Authorization
- Los tiempos de espera (timeout) están configurados en 30 segundos
- Para producción, cambiar las URLs de localhost a las URLs de producción
- Se recomienda usar HTTPS en producción
